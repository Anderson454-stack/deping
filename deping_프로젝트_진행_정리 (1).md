## 1. 프로젝트 개요

### 1.1 프로젝트명

**Deping (디핑)** — Cognitive-Driven Movie Recommendation System (C-DMRS)

### 1.2 한 줄 요약

AI와 대화하면서 내 취향과 오늘의 기분을 알려주면, **나에게 딱 맞는 영화 3편**을 골라주는 서비스.

### 1.3 핵심 가치

기존 추천 시스템은 "스릴러를 본 사용자에게 또 다른 스릴러를 추천"하는 단순한 과거 시청 기록 기반 상관관계에만 의존한다. 디핑은 여기에 사용자가 영화를 어떻게 즐기고 느끼는지 — 즉 **개인별 감상 스타일과 보는 습관**(복잡한 플롯을 좋아하는지, 빠른 전개를 원하는지, 감정 부하를 견디는 정도 등) — 을 함께 고려하여, "스릴러를 좋아하지만 복잡한 플롯은 싫어하는 사용자에게 빠른 전개의 액션 스릴러만 추천"하는 **더 깊이 있고 인간적인 개인화 추천**을 제공한다.

### 1.4 차별화 포인트

- **하이브리드 온보딩**: 대화형 + 카드 선택형을 결합하여 정확한 취향 파악
- **인지적 감상 스타일 반영**: 복잡도 내성, 전개 인내도 등 감상 습관까지 고려
- **CoT 검증 큐레이션**: 추천 후보를 LLM이 논리적으로 검증 후 최종 선별

### 1.5 기존 추천 vs 디핑 추천

| 기존 추천 | 디핑 추천 |
|----------|----------|
| "스릴러를 좋아하시네요" → 테넷 추천 | "스릴러를 좋아하지만 복잡한 건 싫다고 하셨죠" → 존 윅 추천 |
| 장르만 봄 | 장르 + 기분 + 감상 스타일까지 봄 |
| 설문지 형태 | 자연스러운 대화 + 시각적 카드 선택 |

## 2. 시스템 아키텍처 (확정)

### 2.1 확정된 기술 스택

| 항목 | 결정 | 비고 |
|------|------|------|
| 오케스트레이션 | **Python 직접 구현** | openai SDK + 함수 체이닝 |
| Agent 1 (프로파일러) | **GPT-5** | 대화 품질이 핵심이므로 메인 모델 |
| Agent 2 (서처) | **GPT-4.1-mini** | 쿼리 변환은 가벼운 작업, 비용 절약 |
| Agent 3 (큐레이터) | **GPT-5** | CoT 추론 품질이 추천 정확도에 직결 |
| 검색 엔진 | **Azure AI Search** | 벡터 + 시맨틱 하이브리드 |
| 임베딩 | **text-embedding-3-small** | 1536차원 |
| 영화 데이터 | **TMDB API** | 포스터, 예고편, 평점, 출연진 |
| 리뷰 데이터 | **NSMC + IMDB 50K** | 한국어/영문 감성 분석 |
| 프론트엔드 | **Jupyter/Gradio → React** | 1단계 프로토타입 → 2단계 웹 |

### 2.2 오케스트레이션 방식 선정 이유

세 가지 옵션을 비교 후 **Python 직접 구현**으로 결정.

| 비교 항목 | Python 직접 구현 | LangChain | Semantic Kernel |
|----------|-----------------|-----------|-----------------|
| 학습 비용 | 낮음 | 높음 | 높음 |
| 디버깅 | 쉬움 | 어려움 | 보통 |
| Azure 호환 | 좋음 | 좋음 | 최상 |
| Jupyter 호환 | 최상 | 좋음 | 보통 |
| 2주 내 완성 | 충분 | 빠듯 | 빠듯 |

**선정 이유**:
1. 프레임워크 학습에 3~5일 소요 → 2주 중 절반을 학습에 쓰면 구현 시간 부족
2. openai SDK의 Function Calling만으로 3-에이전트 파이프라인 충분히 구현 가능
3. 발표 시 "이 함수가 프로파일러, 이 함수가 서처" 식으로 코드 흐름을 바로 보여줄 수 있음

### 2.3 모델 배치 이유

| 에이전트 | 모델 | 이유 |
|---------|------|------|
| Agent 1 (프로파일러) | GPT-5 | 자연스러운 대화 + 인지 성향 파악에 높은 추론 능력 필요 |
| Agent 2 (서처) | GPT-4.1-mini | 프로필 JSON → 검색 쿼리 변환은 정형화된 작업, 비용 절약 |
| Agent 3 (큐레이터) | GPT-5 | CoT 추론으로 인지 부하 검증 — 사용자 체감 품질에 직결 |

### 2.4 전체 파이프라인

```
사용자 접속
    │
    ▼
┌─────────────────────────────────────────────────┐
│  Agent 1 — 프로파일러 (GPT-5)                      │
│  대화형 질문 + 카드 선택 UI                          │
│  → 취향 + 인지 성향 → 프로필 JSON 출력              │
└────────────────────┬────────────────────────────┘
                     │ 프로필 JSON
                     ▼
┌─────────────────────────────────────────────────┐
│  Agent 2 — 서처 (GPT-4.1-mini)                    │
│  프로필 → 검색 쿼리 변환                             │
│  Azure AI Search (벡터+시맨틱) + TMDB API 보강      │
│  → 후보 15편 출력                                  │
└────────────────────┬────────────────────────────┘
                     │ 후보 15편
                     ▼
┌─────────────────────────────────────────────────┐
│  Agent 3 — 큐레이터 (GPT-5)                        │
│  CoT 추론으로 인지 부하 검증                          │
│  → 최종 3편 선별 + 개인화된 추천 이유 작성             │
└────────────────────┬────────────────────────────┘
                     │ 최종 추천
                     ▼
┌─────────────────────────────────────────────────┐
│  영화 추천 카드 UI                                  │
│  포스터 + 추천 이유 + 예고편 링크                     │
└─────────────────────────────────────────────────┘
```

## 3. 에이전트 구조 비교 (검토 중)

팀 논의가 필요한 사항. 세 가지 구조를 비교 중이며, **C. 하이브리드**를 추천.

### 3.1 Option A — 순차 파이프라인

```
사용자 → 프로파일러 → 서처 → 큐레이터 → 결과
```

| 항목 | 평가 |
|------|------|
| 구현 난이도 | 쉬움 |
| 디버깅 | 쉬움 (순서대로 추적) |
| 응답 속도 | 느림 (직렬 대기) |
| 추천 다양성 | 보통 (단일 관점) |
| 2주 내 구현 | 충분히 가능 |

장점: 안정적, 디버깅 쉬움, 발표 설명 쉬움
단점: 서처가 한 가지 관점으로만 검색 → 추천 다양성 제한

### 3.2 Option B — Expert 병렬 (Grok 스타일)

```
프로파일러
    ↓
장르 전문가 / 감성 전문가 / 트렌드 전문가  (병렬)
    ↓ 각자 후보 제출
종합 판단자 (GPT-5)
```

| 항목 | 평가 |
|------|------|
| 구현 난이도 | 높음 |
| 디버깅 | 어려움 (병렬 결과 추적) |
| 응답 속도 | 빠름 (병렬 호출) |
| 추천 다양성 | 높음 (다중 관점) |
| 2주 내 구현 | 빠듯함 |
| API 비용 | 높음 (병렬 호출 x3) |

장점: 다양한 관점의 추천, 발표 시 인상적
단점: 병렬 호출 관리, 결과 병합 로직, 충돌 해소 추가 복잡도

### 3.3 Option C — 하이브리드 (추천)

```
프로파일러 (GPT-5)
    ↓
메인 서처 (AI Search) + 보조 서처 (감성 분석)  ← 여기만 병렬
    ↓ 후보 병합
큐레이터 + CoT 검증 (GPT-5)
```

| 항목 | 평가 |
|------|------|
| 구현 난이도 | 보통 |
| 디버깅 | 쉬운 편 (병렬은 서처만) |
| 응답 속도 | 빠른 편 |
| 추천 다양성 | 높은 편 (2개 관점) |
| 2주 내 구현 | 가능 |

**추천 이유**:
- 순차 파이프라인의 안정성 + Expert 구조의 다양성을 균형 있게 가져감
- ThreadPoolExecutor로 서처 2개만 병렬 호출 (기존 주식 AI 챗봇 경험 활용 가능)
- 발표: "순차 파이프라인에 검색 단계만 병렬로 최적화했습니다" — 한 줄 설명

### 3.4 팀 논의 필요 사항

- Grok 4.2 Expert 스타일로 완전 병렬 구조를 가져갈지
- 아니면 하이브리드로 서처만 병렬로 할지
- API 비용 대비 추천 품질 향상이 어느 정도인지

## 4. 온보딩 UX 설계 (확정)

### 4.1 결정: 하이브리드 온보딩

**대화형 베이스 + 카드 선택형 혼합**

기분, 장르 선호, 감상 스타일 같은 주관적 항목은 **대화형**으로,
좋아하는 영화, 감독, 배우 같은 고유명사 항목은 **카드 선택형**으로 수집한다.

### 4.2 온보딩 흐름

```
Step 1 (대화형)  "오늘 기분이 어떠세요?"
    → 퀵버튼: 편하게 쉬고 싶어요 / 신나는 게 좋아요 / 감성적인 날이에요
    → 또는 자유 입력

Step 2 (대화형)  "좋아하는 장르가 있으세요?"
    → 퀵버튼: SF / 액션 / 로맨스 / 코미디 / 스릴러

Step 3 (카드형)  "좋아하는 영화가 있다면 골라주세요!"
    → 영화 포스터 카드 그리드 (복수 선택 가능)
    → 인터스텔라, 기생충, 어벤져스, 라라랜드, 올드보이 ...

Step 4 (카드형)  "좋아하는 감독이 있나요?"
    → 감독 카드 그리드 (복수 선택 가능)
    → 봉준호, 크리스토퍼 놀란, 드니 빌뇌브, 박찬욱 ...

Step 5 (카드형)  "배우도 골라볼까요?"
    → 배우 카드 그리드 (복수 선택 가능)

Step 6 (대화형)  "복잡한 이야기를 좋아하세요, 단순한 게 좋으세요?"
    → 퀵버튼: 단순한 게 좋아요 / 복잡해도 OK

Step 7 (대화형)  "영화 볼 때 뭐가 제일 중요하세요?"
    → 퀵버튼: 영상미 / 스토리 / 배우 연기 / 음악
```

### 4.3 세 가지 온보딩 방식 비교 (검토 완료)

| 비교 항목 | 카드 선택형 (스포티파이) | 대화형 (채팅) | 하이브리드 (확정) |
|----------|----------------------|-------------|-----------------|
| 수집 속도 | 빠름 (30초) | 보통 (1~2분) | 보통 (1분) |
| 인지 성향 파악 | 제한적 | 풍부 | 풍부 |
| 고유명사 정확도 | 높음 (선택) | 낮음 (타이핑 오류) | 높음 (선택) |
| 구현 난이도 | 쉬움 | 보통 | 보통 |
| 차별화 | 낮음 (흔한 패턴) | 높음 | 최상 |

### 4.4 구현 방식

프로파일러 에이전트가 대화 도중 Function Calling으로 UI 함수를 호출:

- `show_movie_cards()` → 영화 선택 카드 그리드 팝업
- `show_director_cards()` → 감독 선택 카드 그리드 팝업
- `show_actor_cards()` → 배우 선택 카드 그리드 팝업

사용자 선택 결과는 프로파일러에게 다시 전달되어 프로필 JSON에 반영.

## 5. 에이전트 상세 설계

### 5.1 Agent 1 — 프로파일러

**역할**: 자연스러운 대화 + 카드 선택을 통해 취향과 인지적 감상 스타일을 수집

**수집 항목 — 메타데이터 선호 (기본)**

| 항목 | 설명 | 수집 방식 |
|------|------|----------|
| mood | 현재 기분 / 원하는 분위기 | 대화형 (퀵버튼) |
| genres | 선호 장르 | 대화형 (퀵버튼) |
| tone | 가벼움/무거움 선호 | 대화형 |
| reference | 좋아하는 영화 | 카드 선택형 |
| directors | 좋아하는 감독 | 카드 선택형 |
| actors | 좋아하는 배우 | 카드 선택형 |
| priority | 중시 요소 (영상미, 스토리 등) | 대화형 (퀵버튼) |
| avoid | 회피 요소 | 대화형 |
| era | 선호 시대 | 대화형 |

**수집 항목 — 인지적 감상 스타일 (C-DMRS 확장)**

| 항목 | 설명 | 대화 속 질문 예시 |
|------|------|-------------------|
| patience | 전개 인내도 | "느린 전개도 괜찮으세요?" |
| complexity | 복잡도 선호 | "복잡한 이야기를 좋아하세요?" |
| emotional_load | 감정 부하 내성 | "여운이 오래 남는 영화 좋아하세요?" |
| visual_style | 시각 스타일 선호 | "화려한 영상이 좋으세요?" |
| cognitive_load | 인지 부하 내성 | "자막 읽으면서 복선 찾는 거 괜찮으세요?" |

**프로필 JSON 출력 예시**:

```json
{
  "mood": "relaxed",
  "genres": ["sci-fi"],
  "tone": "light",
  "reference": ["Interstellar", "Gravity"],
  "directors": ["Christopher Nolan"],
  "actors": ["Matthew McConaughey"],
  "priority": "visuals",
  "avoid": ["heavy_drama", "horror"],
  "era": "any",
  "cognitive": {
    "patience": "medium",
    "complexity": "low",
    "emotional_load": "medium",
    "visual_style": "cinematic",
    "cognitive_load": "low"
  }
}
```

### 5.2 Agent 2 — 서처

**역할**: 프로필 JSON → 검색 쿼리 변환 → Azure AI Search 하이브리드 검색 → TMDB 메타데이터 보강

**처리 과정**:

1. 프로필 JSON의 각 필드를 자연어 쿼리 + 필터 조건으로 변환
2. Azure AI Search에서 벡터 검색(줄거리 유사도) + 시맨틱 검색(장르/감독 필터) 실행
3. TMDB API에서 포스터, 예고편 URL, 출연진 정보 보강
4. 후보 약 15편을 Agent 3에게 전달

**하이브리드 옵션 (팀 논의 후 결정)**:
- 보조 서처를 병렬로 추가: NSMC 감성 분석 기반 검색
- ThreadPoolExecutor로 메인 서처 + 보조 서처 병렬 호출

### 5.3 Agent 3 — 큐레이터

**역할**: 후보 15편 + 프로필 JSON → CoT 추론으로 인지 부하 검증 → 최종 3편 선별 + 추천 이유 작성

**CoT 검증 예시**:

```
후보: 테넷 (Tenet, 2020)
- 사용자: complexity=low, cognitive_load=low
- 영화: plot_complexity=5, pacing=medium
- 판정: 인지 부하 불일치 → 제외 ❌

후보: 그래비티 (Gravity, 2013)
- 사용자: visual_style=cinematic, mood=relaxed
- 영화: visual_score=0.95, plot_complexity=2, runtime=91분
- 판정: 전 항목 일치 → 채택 ✅
- 추천 이유: 영상미 + 짧은 런타임 강조
```

**선별 규칙**:
1. 인지적 성향과 영화 속성의 불일치가 있으면 제외
2. 최종 3편은 서로 다른 매력 포인트를 가져야 함 (다양성)
3. 추천 이유는 사용자가 대화에서 사용한 표현을 반영
4. "~하셨으니까", "~좋아하시잖아요" 등 대화체 유지

## 6. 데이터 파이프라인

### 6.1 데이터 소스

| 소스 | 용도 | 규모 |
|------|------|------|
| TMDB API | 영화 메타데이터 | MVP: 2,000~3,000편 |
| NSMC | 한국어 리뷰 감성 분석 | 20만 리뷰 |
| IMDB 50K | 영문 리뷰 감성 분석 | 5만 리뷰 |
| LLM 배치 | 인지적 속성 생성 | 대상 영화 전체 |

### 6.2 영화별 인지적 속성

각 영화에 대해 다음 속성을 사전에 산출하여 인덱스에 포함:

| 속성 | 설명 | 산출 방법 | 예시 |
|------|------|----------|------|
| pacing | 전개 속도 | 장르+런타임 규칙 + LLM 검증 | 존 윅=빠름, 문라이트=느림 |
| plot_complexity | 이야기 복잡도 (1~5) | LLM이 줄거리 읽고 평가 | 마션=2, 테넷=5 |
| emotional_intensity | 감정적 무게 | 리뷰 감성 분석 분산값 | 코미디=낮음, 쉰들러리스트=높음 |
| visual_score | 영상미 점수 | TMDB 인기도 + 촬영상 수상 이력 | 그래비티=높음 |
| sentiment_score | 관객 감성 점수 | NSMC + IMDB 감성 분석 평균 | - |

### 6.3 인덱스 구축 파이프라인

```
TMDB API 수집 (메타데이터)
    │
    ├─→ 줄거리 → text-embedding-3-small → overview_vector (1536차원)
    │
    ├─→ 줄거리 + 장르 → GPT-4.1-mini 배치 → pacing, plot_complexity
    │
    ├─→ NSMC/IMDB 리뷰 매칭 → 감성 분석 → sentiment_score
    │
    └─→ 전체 필드 조합 → Azure AI Search 인덱스 업로드
```

### 6.4 MVP 데이터셋 기준

- IMDB 평점 6.0 이상
- 2000년 이후 개봉
- 한국어 제목 존재 (TMDB ko-KR)
- 주요 장르 커버: 액션, SF, 드라마, 코미디, 로맨스, 스릴러, 공포, 애니메이션

## 9. 평가 지표

### 9.1 시스템 품질

| 지표 | 설명 | 목표 |
|------|------|------|
| 검색 정확도 | 프로필 기반 검색 관련성 | 팀원 교차 검증 |

| 응답 시간 | 전체 파이프라인 소요 시간 | 10초 이내 |
| Groundedness | 추천 영화가 실존하는지 | 100% |

### 9.2 추천 품질

| 지표 | 설명 | 측정 방법 |
|------|------|----------|
| 인지 적합도 | 영화 속성 ↔ 사용자 프로필 일치율 | 자동 계산 |
| 사용자 만족도 | 추천 결과 주관 평가 | 데모 시 설문 (1~5점) |

### 9.3 비교 시뮬레이션

- **Baseline**: 장르 + 평점만으로 추천 (인지 성향 무시)
- **Deping**: 장르 + 평점 + 인지 성향 반영
- **비교**: 추천 목록 중복률, 인지 속성 불일치 건수, 만족도 차이
